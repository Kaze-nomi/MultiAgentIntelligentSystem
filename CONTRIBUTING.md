# Руководство по внесению вклада (Contributing)

Спасибо за ваш интерес к внесению вклада в этот проект! Это руководство поможет вам настроить окружение и понять процесс работы.

## Как начать разработку

1.  **Сделайте форк** репозитория на GitHub.
2.  **Клонируйте** ваш форк локально:
    ```bash
    git clone https://github.com/your-username/project-name.git
    ```
3.  **Переключитесь** в директорию проекта:
    ```bash
    cd project-name
    ```
4.  **Создайте новую ветку** для вашей задачи:
    ```bash
    git checkout -b feature/your-feature-name
    ```

## Настройка окружения

Проект использует Python и Pydantic. Для управления зависимостями используется `pip`.

1.  **Убедитесь, что у вас установлен Python** версии 3.8 или выше.
2.  **Создайте и активируйте виртуальное окружение**. Мы рекомендуем использовать встроенный модуль `venv`:
    ```bash
    # Создание окружения
    python -m venv venv

    # Активация окружения (Linux/macOS)
    source venv/bin/activate

    # Активация окружения (Windows)
    .\venv\Scripts\activate
    ```
3.  **Установите зависимости** из файла `requirements.txt`:
    ```bash
    pip install -r requirements.txt
    ```
4.  **Установите проект в "редактируемом" режиме**. Это позволит вносить изменения в код и сразу видеть их в действии без переустановки:
    ```bash
    pip install -e .
    ```

## Code style guidelines

Мы придерживаемся общепринятых практик в Python-сообществе для поддержания чистоты и читаемости кода.

*   **PEP 8**: Основной стандарт оформления кода.
*   **Black**: Используется для автоматического форматирования кода. Перед отправкой PR запустите:
    ```bash
    black .
    ```
*   **isort**: Используется для сортировки импортов. Запустите перед коммитом:
    ```bash
    isort .
    ```
*   **Pydantic**:
    *   Все модели Pydantic должны быть строго типизированы с использованием аннотаций типов Python.
    *   Используйте `Field` для задания метаданных и валидации полей, где это необходимо.
    *   Предпочитайте `default_factory` для изменяемых значений по умолчанию (например, `default_factory=list`).

## Процесс создания PR (Pull Request)

1.  Перед отправкой PR убедитесь, что ваша ветка **актуальна** с основной веткой (`main` или `master`). Перебазируйте ваши изменения:
    ```bash
    git fetch origin
    git rebase origin/main
    ```
2.  **Запушьте** вашу ветку в ваш форк на GitHub:
    ```bash
    git push origin feature/your-feature-name
    ```
3.  Создайте **Pull Request** из вашей ветки в основную ветку оригинального репозитория.
4.  **Заполните шаблон PR**: дайте ему понятное название и опишите, какие изменения вы внесли и почему. Если PR решает какую-то задачу (issue), добавьте ссылку на нее.

## Правила коммитов (Conventional Commits)

Мы используем соглашение о коммитах для автоматического ведения CHANGELOG и лучшей навигации по истории изменений.

Формат коммита:
```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

**Основные типы:**
*   `feat`: Новая функциональность.
*   `fix`: Исправление ошибки.
*   `docs`: Обновление документации.
*   `style`: Изменения, не влияющие на код (форматирование, отсутствующие точки с запятой и т.д.).
*   `refactor`: Изменение кода, которое не исправляет ошибку и не добавляет функциональность.
*   `chore`: Обновление задач сборки, вспомогательных инструментов, библиотек.

**Пример:**
```
feat(models): add email validation to User model

Added a Pydantic EmailStr validator to ensure the email field
contains a valid email address.

Closes #123
```

## Процесс ревью

1.  Каждый Pull Request должен быть проверен как минимум одним из мейнтейнеров проекта.
2.  Будьте конструктивны в комментариях. Цель ревью — улучшить качество кода, а не найти виноватых.
3.  Автор PR должен рассмотреть все комментарии и внести необходимые изменения.
4.  После одобрения ревью и успешного прохождения всех проверок (если они настроены), мейнтейнер может влить PR в основную ветку.

## Тестирование

В настоящее время в проекте **отсутствует автоматизированная система тестирования**.

Перед отправкой PR убедитесь, что ваши изменения не ломают существующую функциональность. **Протестируйте вручную ключевые сценарии**, которые затронуты вашими изменениями. Если вы добавляете новую функциональность, проверьте её работу в различных условиях.
