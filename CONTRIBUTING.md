# Contributing

Этот документ описывает, как внести вклад в проект. Мы приветствуем вклады от сообщества!

## Как начать разработку

1. Форкните репозиторий на GitHub.
2. Клонируйте форк на свой локальный компьютер: `git clone https://github.com/your-username/project-name.git`.
3. Создайте новую ветку для вашей работы: `git checkout -b feature/your-feature-name`.
4. Внесите изменения и протестируйте их локально.

## Настройка окружения

Проект использует Python и pip для управления зависимостями.

1. Убедитесь, что у вас установлен Python (рекомендуется версия 3.8+).
2. Установите зависимости: `pip install -r requirements.txt` (если файл существует; если нет, установите необходимые пакеты вручную).
3. Для разработки рекомендуется использовать виртуальное окружение: `python -m venv venv` и активировать его: `source venv/bin/activate` (на Windows: `venv\Scripts\activate`).

## Code style guidelines

Код должен соответствовать стандартам Python:

- Следуйте [PEP 8](https://www.python.org/dev/peps/pep-0008/) для стиля кода.
- Используйте инструменты линтинга, такие как `flake8` или `black`, для проверки стиля.
- Добавляйте docstrings к функциям и классам в соответствии с [PEP 257](https://www.python.org/dev/peps/pep-0257/).
- Имена переменных и функций должны быть описательными и на английском языке.

## Процесс создания PR

1. Убедитесь, что ваша ветка обновлена с основной веткой: `git pull origin main`.
2. Запушьте изменения: `git push origin feature/your-feature-name`.
3. Создайте Pull Request (PR) на GitHub, описав изменения и причину.
4. В заголовке PR укажите тип изменения (например, "feat: add new feature").
5. Добавьте ссылки на связанные issues, если применимо.

## Правила коммитов (Conventional Commits)

Используйте формат Conventional Commits для сообщений коммитов:

- `feat: описание новой функциональности`
- `fix: исправление бага`
- `docs: изменения в документации`
- `style: изменения стиля кода (не влияющие на логику)`
- `refactor: рефакторинг кода`
- `test: добавление или изменение тестов`
- `chore: прочие изменения (обновление зависимостей и т.д.)`

Пример: `feat: add user authentication module`

Сообщения должны быть на английском языке и описывать изменения кратко.

## Процесс ревью

1. После создания PR назначьте ревьюеров (maintainers проекта).
2. Ревьюеры проверят код на соответствие code style, логику и отсутствие багов.
3. Обсудите комментарии в PR и внесите исправления.
4. После одобрения PR будет слит в основную ветку.

## Тестирование

Проект не имеет реализованной системы тестирования. Если вы добавляете новую функциональность, рассмотрите возможность написания unit-тестов с использованием `unittest` или `pytest`. Убедитесь, что изменения не ломают существующий код. Для запуска тестов (если они будут добавлены) используйте `python -m unittest` или `pytest`.
