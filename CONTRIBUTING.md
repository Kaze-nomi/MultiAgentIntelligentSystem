# Contributing

Этот документ описывает, как внести вклад в проект. Мы приветствуем вклады в виде исправлений багов, новых функций, улучшений документации и т.д.

## Как начать разработку

1. Ознакомьтесь с [README.md](README.md) для понимания проекта.
2. Проверьте открытые issues и pull requests на GitHub, чтобы избежать дублирования.
3. Создайте форк репозитория и клонируйте его на свой компьютер.
4. Создайте новую ветку для ваших изменений: `git checkout -b feature/your-feature-name`.

## Настройка окружения

1. Убедитесь, что у вас установлен Python 3.8+.
2. Установите зависимости с помощью pip:
   ```
   pip install -r requirements.txt
   ```
   Если файла `requirements.txt` нет, установите проект в режиме разработки:
   ```
   pip install -e .
   ```
3. Для работы с Pydantic убедитесь, что все модели и валидации соответствуют документации Pydantic.

## Code style guidelines

- Следуйте PEP 8 для стиля кода Python.
- Используйте инструменты линтинга, такие как flake8 или black, для проверки стиля.
- Для моделей Pydantic используйте аннотации типов и валидаторы согласно официальной документации Pydantic.
- Комментарии должны быть на русском языке, если проект ориентирован на русскоязычную аудиторию.

## Процесс создания PR

1. Убедитесь, что ваши изменения соответствуют code style guidelines.
2. Напишите тесты, если применимо (см. секцию Тестирование).
3. Создайте pull request (PR) из вашей ветки в основную ветку проекта.
4. В описании PR укажите:
   - Что было изменено.
   - Почему это изменение необходимо.
   - Ссылки на связанные issues.
5. Убедитесь, что все CI/CD проверки проходят (если настроены).

## Правила коммитов (Conventional Commits)

Используйте формат Conventional Commits для сообщений коммитов:

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

Типы:
- `feat`: новая функция
- `fix`: исправление бага
- `docs`: изменения в документации
- `style`: изменения стиля (без влияния на логику)
- `refactor`: рефакторинг кода
- `test`: добавление или исправление тестов
- `chore`: изменения в инфраструктуре

Примеры:
- `feat: добавить валидацию email в модели User`
- `fix: исправить ошибку в парсинге данных Pydantic`

## Процесс ревью

1. После создания PR назначьте ревьюеров (maintainers проекта).
2. Ревьюеры проверят код на соответствие стандартам, логику и потенциальные баги.
3. Если есть замечания, исправьте их и обновите PR.
4. После одобрения ревью PR будет слит в основную ветку.

## Тестирование

В проекте тестирование не реализовано (N/A). Если вы добавляете новую функциональность, рассмотрите возможность добавления unit-тестов с использованием pytest, но это не обязательно для текущего вклада. Если тесты будут добавлены в будущем, они должны покрывать модели Pydantic и логику валидации.
